# Implementation Plan: Agent Runner Server and PWA System

**Branch**: `001-agent-runner-server-pwa` | **Date**: 2026-03-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-agent-runner-server-pwa/spec.md`

## Summary

Build a Node.js server that manages sandboxed `claude` CLI processes for autonomous task execution and interactive spec-kit SDD workflows, paired with a Preact PWA for mobile monitoring. The server uses `systemd-run --user --scope` for filesystem isolation, JSONL files for session logs, `ws` for real-time streaming, and `web-push` for push notifications. Agent-framework markdown task files are the source of truth вЂ” the server reads them on demand and auto-loops task runs until all tasks are complete or input is needed. New projects are created through an interactive spec-kit workflow (specify в†’ clarify в†’ plan в†’ tasks в†’ analyze), then implemented autonomously via `run-tasks.sh`.

## Technical Context

**Language/Version**: TypeScript on Node.js 22 (via Nix flake, compiled with `tsc`, dev with `tsx`)
**Primary Dependencies**: `ws` (WebSocket), `web-push` (push notifications), `pino` (structured logging), `tsx` (dev), `preact` (UI), `esbuild` (client build)
**Storage**: Filesystem only вЂ” `~/.agent-runner/projects.json`, per-session `meta.json` + `output.jsonl`
**Testing**: Node.js built-in test runner (`node:test`) + `node:assert`
**Target Platform**: NixOS (Linux), accessed via local network PWA
**Project Type**: Web service (server) + PWA (client)
**Performance Goals**: Voice delivered <5s, notifications <30s
**Constraints**: Single-user, local network, no auth, sandbox via systemd-run --user
**Scale/Scope**: ~5-10 concurrent projects, 1 user, ~8 PWA screens

## Key Design Decisions

### 1. Sandbox Command Chain

```bash
systemd-run --user --scope \
  --property='ProtectHome=tmpfs' \
  --property='BindPaths=<project-dir>' \
  --property='ProtectSystem=strict' \
  --property='NoNewPrivileges=yes' \
  nix develop <project-dir> --command \
  claude --task-file <task-file> --prompt-file <prompt-file>
```

When `systemd-run` is unavailable, the server refuses to start sessions unless both gates are satisfied: (1) server started with `ALLOW_UNSANDBOXED=true` env var, AND (2) session start request includes `allowUnsandboxed: true`. Unsandboxed execution falls back to `nix develop ... --command claude ...` with a visible warning in server logs and session output.

### 2. Task-Run Auto-Loop

```
Start session в†’ spawn claude process в†’ process exits в†’
  parse task file в†’
    unchecked tasks remain, no [?] в†’ spawn new claude process (loop)
    [?] task found в†’ transition to waiting-for-input, notify user
    all tasks [x] or [~] в†’ mark session completed, notify user
    process crashed в†’ mark session failed, notify user
```

Each loop iteration is a separate process spawn (not a long-running process). This keeps process isolation clean and makes crash recovery straightforward.

### 3. Session Log Replay

- Writer: `fs.createWriteStream(path, { flags: 'a' })` вЂ” append mode, one JSONL entry per `write()` call
- Reader (full): `fs.createReadStream(path)` + `readline`
- Reader (offset): `fs.createReadStream(path, { start: byteOffset })` + `readline`
- Client tracks `lastSeq` from WebSocket messages; sends it as query param on reconnect

### 4. PWA Architecture

- Preact with JSX, built with esbuild (source in `src/client/`, output to `public/`)
- Hash-based routing (`#/`, `#/projects/:id`, `#/sessions/:id`)
- Preact functional components with hooks
- Service worker handles push display + basic offline caching of previously viewed logs

### 5. Voice Input

- Default: Browser-native Web Speech API (`webkitSpeechRecognition`)
- Alternative: Google Speech-to-Text API вЂ” toggled via settings page or long-press mic icon
- Transcription happens entirely client-side for Web Speech API
- For Google STT: client records audio в†’ sends to server в†’ server proxies to Google Speech-to-Text API в†’ returns text
- Server needs a `/api/voice/transcribe` endpoint only for the cloud backend
- `GOOGLE_STT_API_KEY` env var configures the cloud backend; health endpoint exposes `cloudSttAvailable`

### 6. Spec-Kit Workflow Orchestration

New projects are created through the full spec-kit SDD workflow:

```
User provides repo name + idea description в†’
  create $AGENT_RUNNER_PROJECTS_DIR/<repo-name>/ в†’
  specify (interactive session) в†’
  clarify (interactive session) в†’
  plan (interactive session) в†’
  tasks (interactive session) в†’
  analyze (interactive session вЂ” resolve issues) в†’
  apply remediations в†’
  user approves в†’
  launch run-tasks.sh (autonomous implementation)
```

Each phase runs as a separate agent session (own context window) to avoid context overload. The user participates interactively in all planning phases. Only the final implementation via `run-tasks.sh` is autonomous.
